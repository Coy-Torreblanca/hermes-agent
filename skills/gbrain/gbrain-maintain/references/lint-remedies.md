# Lint Issue Triage Guide

Reference for interpreting and acting on `gbrain lint` output during brain maintenance sessions.

## The Big Picture

gbrain lint (at `/opt/gbrain/src/commands/lint.ts`) scans every `.md` file on disk with 8 hard-coded rules. It reads **from disk**, not from the brain DB. **95%+ of issues are structurally benign** — they come from auto-imported project documentation files that were never meant to be first-class brain pages.

**The lint tool has zero configuration:** no ignore paths, no per-source rule toggles, no `.gbrainignore` support. Files starting with `.` or `_` are the only filtering.

## Issue Categories

### Group A: Fixable by `--fix` (auto-resolvable)

| Rule | What It Fixes | Prevalence |
|------|--------------|-----------|
| `llm-preamble` | Strips "Of course! Here is..." artifacts | Rare — only on agent-written pages |
| `code-fence-wrap` | Removes wrapping ` ```markdown...``` ` fences | Rare |
| `frontmatter-nested-quotes` | Wraps description values in single quotes | Rare — typically 1 per brain |

**Action:** `gbrain lint <brain_dir> --fix` — this is what the dream cycle does during its lint phase. Usually clears 1-5 issues.

### Group B: Structural/Benign (ignore on project docs)

| Rule | Count (typical brain with repo) | Why Benign |
|------|-------------------------------|-----------|
| `no-frontmatter` | ~510 (50% of issues) | READMEs, CONTRIBUTING, RELEASE notes, AGENTS.md — project boilerplate that has no frontmatter by design |
| `missing-type` | ~509 | Files that happen to have some frontmatter but lack `type:` — autogenerated template frontmatter without editorial intent |
| `missing-created` | ~509 | Same files — no creation date on a file that was git-cloned, not authored |
| `missing-title` | ~203 | Files with frontmatter but no `title:` field |
| `placeholder-date` | ~145 | Template files with `YYYY-MM-DD` unfilled placeholders |
| `empty-section` | ~13 | Markdown stubs with `## Heading` and nothing underneath |

**Action for group B:** These are not real problems. The files index, embed, and search correctly regardless. Do not add frontmatter to 500+ project READMEs — that's a losing battle and provides zero semantic value.

**Workaround:** If the lint noise is distracting, rename heavily affected directories to start with `_` (e.g., `projects/` → `_projects/`). Lint's `collectPages()` skips underscore-prefixed directories. The brain DB and search are unaffected — sync still picks them up.

### Group C: Actionable (fix these)

| Rule | Count | What to Do |
|------|-------|-----------|
| `frontmatter-yaml-parse` | ~5 | YAML syntax errors — fix immediately. Most common cause: `[[wikilinks]]` in `sources:` field without quotes. Fix: `sources: '[[slug-a]], [[slug-b]]'` (single-quote the whole value). |
| `broken-citation` | ~5 | Unclosed `[Source: ...]` bracket — find and close it. The dream cycle auto-fix handles some of these. |
| `empty-section` | ~13 | If the section matters, fill it in. If it's a boilerplate stub, delete the heading. |

**Action for group C:** These should be triaged during a maintenance session. Each one represents a genuine quality issue.

## The "Score Ceiling" Trap

When >80% of brain pages are auto-imported project docs (READMEs, AGENTS.md files):
- **Link density** is structurally capped — project docs have no `[[wikilinks]]`
- **Orphan count** is structurally inflated — all 960 project pages are "orphans" by definition
- **Timeline coverage** is structurally capped — project docs have no timeline entries
- **Lint count** is structurally inflated

**Do not chase these.** The health score will never exceed ~60% when 83% of pages are project docs. This is a correctable upstream limitation (waiting for `lint.ignore_paths` or per-source health scoring), not a maintenance problem.

## The `--fix` Limitation

```
gbrain lint <dir> --fix
```
Only fixes LLM preamble artifacts and code-fence-wrap. It does NOT fix:
- `frontmatter-yaml-parse` (YAML syntax errors)
- `frontmatter-nested-quotes` (nested quotes)
- `no-frontmatter`, `missing-*`, `placeholder-date`, `broken-citation`, `empty-section`

The `[fixable]` tag in the output refers only to the auto-fix capability. Anything tagged `[fixable]` WILL be fixed by `--fix`. Anything not tagged `[fixable]` requires manual intervention.

## Migration & Lint Interaction

The dream cycle's lint phase runs `gbrain lint <brain_dir> --fix` as part of the maintenance pipeline. This clears the 1-5 auto-fixable issues each cycle. The 1,900+ structural issues persist because they're not auto-fixable — not because the dream cycle is broken.
